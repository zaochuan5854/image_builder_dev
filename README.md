# DockerHub 自動ビルド＆デプロイ ワークフロー

このリポジトリは、GitHub上のソースコード変更をトリガーにDockerイメージを自動ビルドし、DockerHubへ配信する GitHub Actions ワークフローです。

パケットロスが多い回線環境では、機械学習ライブラリなど大容量の依存関係を含むDockerイメージをローカルでビルドすると、依存パッケージの取得に時間がかかり、ビルドが失敗しやすくなります。

本ワークフローでは、依存関係の取得とビルドをGitHub Actionsの高速なネットワーク上で実行し、ローカルでは完成したDockerイメージというシリアライズ済みの成果物を取得するだけで利用できます。これにより、ローカル回線への負荷を抑えつつ、安定してDockerイメージを利用できます。

## 主な機能

* **自動ビルド＆プッシュ**: `main` ブランチおよび `v*.*.*` タグでDockerイメージを自動ビルド・プッシュします。
* **PRビルド**: Pull Requestではビルド検証のみを実行し、レジストリへのログインやプッシュは行いません。
* **圧縮形式の選択**: `gzip`、`zstd`、`estargz` をサポートします。
* **通知機能**: `ntfy.sh` にビルド結果を通知します。

---

## 💡 想定されるユースケース

* **不安定な回線環境での開発**
  依存関係の取得とビルドをGitHub Actionsへオフロードし、ローカルでは完成したDockerイメージのみを取得します。

* **マルチアーキテクチャ向けビルド**
  x86_64・ARM向けイメージを単一のワークフローでビルドできます。

* **安全なDockerHub運用**
  タグの上書き制御や通知機能により、誤操作やビルド失敗による影響を抑えられます。

## 必要な事前設定 (GitHub Secrets & Variables)

動作させるには、GitHubリポジトリの設定（`Settings` > `Secrets and variables` > `Actions`）にて、以下の **Secrets (機密情報)** および **Variables (一般変数)** を追加する必要があります。

| 設定名 | 設定場所 | 必須/任意 | 説明 |
| --- | --- | --- | --- |
| `DOCKERHUB_USERNAME` | **Secrets** | **必須** | DockerHubのユーザー名。 |
| `DOCKERHUB_TOKEN` | **Secrets** | **必須** | DockerHubの Personal Access Token (PAT)。イメージのプッシュおよびタグの削除権限が必要です。 |
| `DOCKERHUB_REPO_NAME` | **Variables**<br>(Secretsも可) | *任意* | プッシュ先となるDockerHubのリポジトリ名。省略した場合は、GitHubリポジトリ名が自動的に小文字化されて使用されます。（両方にある場合は Variables が優先されます） |
| `OVERWRITE` | **Variables** | *任意* | **`1`** に設定すると、重複する既存タグの自動削除とクリーンな上書きプッシュが有効になります。<br>未設定または **`0`**（デフォルト）の場合、既に同じタグがDockerHubに存在すると**ビルドをエラーで終了**させ、上書きを防止します。 |
| `NTFY_URL` | **Secrets** | *任意* | `ntfy.sh` の通知受信用URL（例: `https://ntfy.sh/your_private_topic`）。設定すると、成否の結果がプッシュ通知されます。 |
| `PLATFORMS` | **Variables** | *任意* | ビルド対象のプラットフォーム。カンマ区切りで複数指定可能（例: `linux/amd64,linux/arm64`）。<br>未設定時は **`linux/amd64`** が適用されます。 |
| `COMPRESSION` | **Variables** | *任意* | イメージレイヤーの圧縮アルゴリズム。 `gzip`（デフォルト）、`zstd`、または `estargz` を指定できます。 |

## セットアップ手順

1. **ワークフローファイルの追加**
このリポジトリの `.github/workflows/build_push.yml` をご自身のリポジトリの同じパスへ配置します。
2. **Dockerfileの配置**
リポジトリのルートディレクトリに、ビルド対象となる `Dockerfile` を配置します。
3. **Secrets & Variablesの設定**
上述の [GitHub Secrets & Variables](#必要な事前設定-github-secrets--variables) を設定します。
4. **ワークフローの実行**
`main` ブランチへのコミットプッシュや、Pull Requestを作成すると、自動的に GitHub Actions がトリガーされます。

## ワークフローの挙動詳細

### トリガーごとの動作マトリクス

| Gitイベント | PR 時 | main Push時 | リリースタグ Push時 (`v*.*.*`) |
| --- | --- | --- | --- |
| **ビルド処理** | 実行する | 実行する | 実行する |
| **レジストリログイン** | スキップ | 実行する | 実行する |
| **重複タグの事前削除** | スキップ | 実行する (`OVERWRITE=1`時)<br>**重複時にエラー終了** (`OVERWRITE=0`時) | 実行する (`OVERWRITE=1`時)<br>**重複時にエラー終了** (`OVERWRITE=0`時) |
| **イメージのプッシュ** | スキップ | 実行する | 実行する |
| **付与されるタグ** | `main` (検証用)<br>`sha-xxxxxxx` | `main`<br>`latest`<br>`sha-xxxxxxx` | `1.0.0`<br>`1.0`<br>`1`<br>`latest`<br>`sha-xxxxxxx` |

## ntfy.sh による結果通知

`NTFY_URL` を設定している場合、ワークフローの成否に関わらず（ビルド失敗時や上書きエラー発生時でも）最後のステップで `ntfy.sh` に対して通知が送信されます。

## 📶 帯域制限（低帯域）環境向けの推奨設定

本プロジェクトは、「ローカル環境のインターネット帯域が狭い（またはモバイル回線等の従量課金制である）」ユーザーが、重い依存関係（Node.js の node_modules や Python の pip パッケージ等）のダウンロードやコンパイルを GitHub Actions 側にすべて委ね、最終的に最適化・圧縮された最小限の Docker イメージのみをローカルにプルして高速利用することを目指しています。

このユースケースにおいて、ダウンロードサイズや転送帯域を最小化するための推奨設定は以下の通りです。

### 1. 圧縮方式に `zstd` を指定する

イメージのダウンロードサイズを極限まで縮小し、ローカルでの展開処理を高速化するため、`COMPRESSION` を **`zstd`** に設定することを強く推奨します。

### 2. 対象プラットフォームをローカル環境のみに限定する

不要なアーキテクチャ向けのバイナリ転送を防ぐため、`PLATFORMS` にはご自身のローカル環境のアーキテクチャ（通常は `linux/amd64` のみ、または M1/M2 Mac などの場合は `linux/arm64` のみ）を明示的に指定します。

### 3. Dockerfile でのマルチステージビルドの徹底

ビルド用ツール（コンパイラ、SDK、一時キャッシュ等）が最終イメージに残らないよう、`Dockerfile` でマルチステージビルドを使用し、最終実行用のステージ（`alpine` や `distroless` など）は極力軽量な構成にしてください。
